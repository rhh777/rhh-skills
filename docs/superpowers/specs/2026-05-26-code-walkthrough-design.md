# 设计文档:code-walkthrough skill

> 日期:2026-05-26
> 目标:一套帮助人**理解 AI 生成代码**的对话式讲解/确认 skill。通过结构化引导 + 逐项确认,
> 让用户按模块/函数粒度建立心智模型,确认实现是否符合预期,并沉淀理解笔记与存疑清单。

## 背景与定位

当前工具大多做两件事:**生成代码**(Copilot/Cursor/aider)和**自动审查找 bug**(PR-Agent/CodeRabbit)。
本 skill 填补第三种被忽视的环节:**让人真正"理解"AI 写的代码**——不是替你判断对错,而是带你建立 mental model。

本质是一个**苏格拉底式的代码讲解 + 确认教练**。

- **主场景**:事后理解(接手 AI 生成的陌生代码库 / 大项目)
- **次场景**:边写边审(AI 刚写完一段立即讲解确认)
- **驱动方式**:混合——AI 建全景并给出建议讲解路线,用户随时可打断、跳转、追问

## 核心交互原则

1. **一次只下钻一段,讲完必停**——这是"逐项确认"的命根子,不一口气讲完整个模块。
2. **用户随时可跳转/打断**(混合模式):引擎能放下当前路线去响应,之后能回到原路线。
3. **确认按固定优先级组织提问**:
   - ① **对齐设计意图**(主线):"这里用了乐观锁处理并发,你期望的是这个策略吗?"
   - ③ **标记存疑点**(补充):AI 主动指出可疑/有风险/不符常规处,请用户判断
   - ④ **陌生技术扫盲**:遇到用户可能不懂的技术,先讲清再继续(框架包供料)
   - ② **反向检验理解**(可选):用提问检查用户是否听懂

## 整体结构

```
code-walkthrough/
├── SKILL.md                      # 引擎:对话流程 + 确认机制 + 笔记规则(不认识任何具体技术栈)
├── references/
│   ├── note-template.md          # 理解笔记 + 存疑清单格式模板
│   └── packs/                    # 可插拔燃料,按探测到的技术栈"按需加载",不进上下文
│       ├── languages/            # go.md, python.md …
│       ├── frameworks/           # spring.md, react.md …
│       ├── systems/              # es.md, volcano.md …(基础设施/中间件)
│       └── projects/             # opencompass.md …(具体开源项目的脉络导览)
└── (理解笔记产出到用户项目内,不放在 skill 目录)
```

**分层**:
- `SKILL.md` = 不变的引擎(流程、确认、沉淀规则)
- `references/packs/**` = 可插拔燃料,引擎探测技术栈后按需读入
- 笔记沉淀到**用户项目内**(`docs/walkthrough/`),跨会话累积成项目地图

## 对话流程引擎

```
[开场] 确定范围
   ├─ 用户指定模块/文件? → 直接进入
   └─ 整个项目? → 探测技术栈 + 入口
                  └─ 大项目? → 可选:派 subagent 跑全景扫描,产出"项目地图"

[加载 pack] 探测技术栈,匹配并加载所有相关 pack(可多个叠加:语言+框架+系统+项目)
            无匹配 → 退化为纯通用讲解(完全可用)

[建全景] 对目标模块给出:职责一句话 / 对外接口 / 依赖 / 主链路入口
   └─ 给一条"建议讲解路线"(按主链路排序的函数/步骤清单)

[下钻循环] 沿路线逐段讲解,每段:
   1. 讲这段做什么(必要时按【④扫盲】展开陌生技术 ← pack 供料)
   2. 讲完必停,按优先级 ①③④② 对齐:
      ① 实现是否符合预期?(对齐设计意图)        ← 主线
      ③ 我觉得 X 可疑/有风险,你怎么看?          ← 补充(pack 的"常见存疑点"供参考)
      ④ (遇陌生技术)先扫盲再继续
      ② (可选)反向提问检验理解
   3. 用户响应 → 确认 / 追问 / 跳转其它模块(随时可打断)

[沉淀] 每聊完一个模块 → 写/更新理解笔记 + 把存疑点追加到清单
```

## 产出物

### 理解笔记(每模块一份)

产出到 `docs/walkthrough/<模块名>.md`,**增量更新**(二次讲解合并,不覆盖):

```markdown
# <模块名> 理解笔记
> 最后更新:YYYY-MM-DD | 技术栈:<探测到的栈>

## 一句话职责
<这个模块是干什么的>

## 对外接口 / 入口
<别人怎么用它,从哪进>

## 依赖
<它依赖谁、谁依赖它>

## 主链路
1. 入口 X → 调用 Y → 落到 Z
   - 关键实现:<用了什么策略,为什么>
   - ✅ 已确认意图:<用户确认过的设计预期>

## 扫盲记录
- <这次搞懂的陌生技术/模式,一句话留存>
```

### 存疑清单(全局一份)

`docs/walkthrough/_open-questions.md`,带状态流转(开放 / 已确认无碍 / 待追改 / 已解决),
作为用户后续"追 AI 改"的待办:

```markdown
# 存疑清单
| # | 模块 | 位置 | 疑点 | 类型 | 状态 |
|---|------|------|------|------|------|
| 1 | auth | TokenService:42 | 并发下乐观锁可能丢更新 | 风险 | 待追改 |
| 2 | order | 全局 | 没看到超时处理 | 待确认 | 开放 |
```

## Pack 格式(可插拔燃料)

所有 pack 用统一结构,引擎探测到技术栈后按需读入。示例 `references/packs/frameworks/spring.md`:

```markdown
# Spring Boot 讲解包

## 触发探测
pom.xml / build.gradle 含 spring-boot;或 @SpringBootApplication

## 重点看什么(检查清单)
- Bean 生命周期与注入方式(构造器 vs 字段注入)
- 事务边界:@Transactional 的传播级别、自调用失效陷阱
- AOP 切面影响了哪些方法
- 配置来源:application.yml / @ConfigurationProperties
- 请求链路:Controller → Service → Repository 分层是否清晰

## 常见存疑点(供"③标记存疑"参考)
- @Transactional 加在 private/自调用方法上 → 不生效
- 字段注入导致循环依赖被悄悄吞掉
- N+1 查询(JPA 懒加载)

## 扫盲套路(供"④扫盲")
- 依赖注入:用大白话讲"谁负责 new 对象"
- 代理机制:为什么自调用事务失效
```

### Pack 分类与叠加

- **languages / frameworks / systems**:通用知识,跨项目复用
- **projects**(如 OpenCompass):针对某个具体仓库的"地图导览"——核心抽象、配置怎么串、扩展点在哪
- **关键机制:引擎支持同时加载多个 pack 并合并**(语言 + 框架 + 系统 + 项目一起喂),
  而非只匹配一个。新增技术栈只需往对应子目录丢一个 md,引擎零改动。

## 范围(首发)

- 实现通用引擎(SKILL.md)+ note-template + **仅 Spring 一个 pack**
- 其它 pack(go / es / volcano / opencompass …)以后按需加
- 无匹配 pack 时引擎退化为纯通用讲解,保证开箱可用

## 非目标(YAGNI)

- 不做自动 bug 检测/打分(那是 code review 工具的活)
- 不做独立的每语言 skill(方案 B)——pack 多到需要单独分发时再演进
- 不强制全景扫描——仅作为大项目的可选开场步骤
```
